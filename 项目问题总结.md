# 项目问题总结

> 工业级智能 Agent 平台开发过程中遇到并修复的核心问题汇总

---

## 一、环境与配置问题

### 1. conda 虚拟环境

- **问题**：`pip install` 需要 escalated 权限，无法在基础环境安装
- **解决**：新建 `crafto` 虚拟环境 `conda create -n crafto python=3.11`
- **当前依赖**：fastapi, uvicorn, sqlalchemy, openai, httpx, numpy, pydantic, qdrant-client, redis

---

## 二、前端问题

### 2. 输入框无法使用（disabled)

- **现象**：页面加载后输入框灰色不可输入，发送按钮灰色不可点击
- **根因**：HTML 中 textarea 和 send-btn 硬编码了 `disabled` 属性，且 `showApp()` 登录后没有主动启用
- **修复**：
  - 移除 HTML 中的 `disabled`
  - 新增 `enableChatInput()` 函数，在 `showApp()` 末尾调用
  - 启用输入框、发送按钮、自动聚焦、状态指示器

### 3. getAuthHeaders 箭头函数导致 this 丢失

- **现象**：所有 API 请求返回 401（POST 发消息成功但 GET 会话列表失败）
- **根因**：`API.headers` 定义为箭头函数 `() => ({...})`，在对象字面量中，箭头函数捕获的是全局 `this`（严格模式下为 `undefined`），虽然没直接拿 `this` 但方法绑定链路被打破
- **修复**：抽取为独立顶层函数 `getAuthHeaders()`，所有 fetch 调用统一使用

### 4. 浏览器 localStorage 残留旧认证

- **现象**：清除测试数据库后，旧 user_id 仍在 localStorage 中，每次加载页面都用过期 token 请求 401
- **修复**：`loadSessions()` 捕获到 401 错误时自动 `store.clear()` 并跳转回登录页

### 5. 切换会话不加载历史消息

- **现象**：点击"新对话"后再点回旧会话，聊天区空白
- **根因**：`switchSession()` 只清空了聊天区，没有从服务器拉取历史消息
- **修复**：
  - 新增 `GET /api/v1/chat/sessions/{id}/messages` 接口
  - 新增 `SessionManager.get_messages()` 方法
  - `switchSession()` 调用 API 加载历史消息并渲染

### 6. 模板字面量语法错误

- **现象**：所有 JS 功能失效（登录、注册全部无响应），浏览器控制台报语法错误
- **根因**：Python heredoc 中 `\`` 被原样写入 .js 文件，JavaScript 不认 `\`` 作为模板字面量起始符
- **修复**：`\` → `` ` ``、`\$` → `$`

---

## 三、认证架构问题

### 7. X-API-Key 优先级高于 Bearer Token

- **现象**：注册/登录成功（200），但紧接着的 API 调用全部 401
- **根因**：`getAuthHeaders()` 同时发送 `X-API-Key`（非空）和 `Authorization` 头。中间件的优先级逻辑：
  ```python
  if api_key:                              # X-API-Key 永远非空 → 永远走这
      user_id = self._resolve_api_key(...)   # 但哈希格式不对，永远 None
  elif auth_header.startswith("Bearer "):    # ← 没机会执行！
      user_id = self._resolve_token(...)
  ```
- **修复**：
  - `elif` → `if not user_id and`（_resolve_api_key 失败后 fallback 到 _resolve_token）
  - `_resolve_api_key` 中的 `str(user.created_at)` 改为 `user.created_at.isoformat()`（对齐生成时的格式）

## 四、Agent ReAct 循环问题

### 8. DeepSeek tool_calls 未被提取

- **现象**：问天气时 Agent 只回复"我查一下"但不触发真正的工具调用
- **根因**：`LLMResponse` 没有 `tool_calls` 字段，`DeepSeekClient.chat()` 只提取了 `choice.message.content` 丢了 `choice.message.tool_calls`
- **修复**：
  - `LLMResponse` 新增 `tool_calls: Optional[list[dict]]`
  - `deepseek.py` 提取原生 tool_calls 并传入 LLMResponse
  - `agent.py` 的 `_react_loop` 优先使用 `response.tool_calls`

### 9. tool_calls 存入记忆后格式不兼容

- **现象**：DeepSeek 返回 400：`messages[2]: missing field 'type'`
- **根因**：`_react_loop` 将 tool_calls 从 OpenAI 格式转换为简化格式，但第二轮 API 调用发回 DeepSeek 时需要完整格式
- **修复**：保留 OpenAI 完整格式（含 `type: "function"`、`function: {name, arguments}`）存入记忆，仅提取 `name` + `arguments` 用于执行

### 10. 最终回复携带错误格式的 tool_calls

- **现象**：多轮对话时 DeepSeek 报 400：`missing field id at line 1 column 2307`
- **根因**：`process_message()` 将工具执行结果 `[{"tool_name": "...", "success": true}]` 作为 `tool_calls` 传入最终回复消息，下一轮对话 DeepSeek 解析失败
- **修复**：去掉 `tool_calls=tool_results`，工具结果已在 ReAct 循环中以独立 tool 消息存储

### 11. add_assistant_message 在工具循环内重复执行

- **现象**：多工具调用时报 400：`insufficient tool messages following tool_calls message`
- **根因**：`for idx, tc in enumerate(tool_calls_in_response)` 循环内每次都调用 `add_assistant_message`，导致同一个 tool_calls 被多次写入历史，但工具响应消息数量不匹配
- **修复**：将 `add_assistant_message` 移到循环外，只存一次

## 五、后端基础设施

### 12. 依赖自动降级

- **Redis** 不可用时自动降级到 `_InMemoryRedis`
- **Qdrant** 不可用时自动降级到 `_InMemoryVectorStore`

### 13. Agent 陷入死循环，耗尽可能迭代次数

- **现象**：针对多步操作（如瑞幸咖啡下单：查门店→搜商品→预览→下单），LLM 不断调用新工具，直到跑满 `max_iterations` 后返回无意义的兜底消息"我已经完成了多轮分析和操作..."
- **根因**：
  - 下单流程需要 4-5 次工具调用，但初始 `max_iterations=5` 刚好不够
  - LLM 调用完最后一步后仍会继续调用无关工具（如 search），继续消耗迭代
  - 兜底消息没有汇报已完成的进度，而是给出无意义提示
- **修复**：
  - `max_iterations` 从 5 → 15（给足迭代空间）
  - 新增**搜索循环检测**：最近 4 次调用中 3 次以上 search → 自动终止
  - 替换兜底消息为**结果摘要**：把已完成的工具调用用图标和文字汇总，而不是"任务处理超时"
  - 暂时禁用 `search` MCP 服务（防止 LLM 在订购流程中回退到搜索）

### 14. LLM 对问题调用的 MCP 工具不准确

- **现象**：问"广州天气"却调用 `search__search_web`（而不是 `weather__get_current_weather`）；问"南昌位置"却调用 `search__search_web`（而不是 `geo__get_city_location`）
- **根因**：
  - 工具列表太长（21+ 个工具），LLM 注意力优先集中在靠前的工具
  - `search__search_web` 的名字和描述太泛化，LLM 认为什么都能搜到
  - 系统提示词没有明确告诉 LLM 工具的选择优先级
  - `service__tool` 前缀的命名方式 LLM 不熟悉，需要额外解释
- **修复**：
  - **工具排序**：`mcp_manager.get_openai_tools()` 按优先级输出（luckin → weather → geo → sandbox → search），行动类工具排在最前
  - **工具描述**：search 工具加 `[只读]` 标签，描述明确写上"不能下单/查天气/执行操作"
  - **系统提示词**：
    - 新增 `MCP 工具（第三方远程服务）` 章节，列出每个服务的用途
    - 新增 `工具选择优先级（从高到低）` 排序指引
    - 新增 `下单/购买类操作必须使用对应的 MCP 工具` 规则
    - 新增 `只读搜索不能用于下单、查询数据库、执行代码等操作` 规则
  - **坐标精度优化**：`geo__get_city_location("黄埔区")` 比 `geo__get_city_location("广州市")` 更精确，避免 Luckin 门店查询因坐标偏移而返回空数据
  - **临时禁用 search**：在调试期间注释掉 search-mcp，强制 LLM 使用正确的专用工具
